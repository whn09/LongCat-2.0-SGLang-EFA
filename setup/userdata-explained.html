p5en 节点 UserData 说明——开机自动做了什么

适用于 www-h200-v12 CloudFormation / Terraform 模板 · 每台节点开机自动执行 · 全程无需人工干预
🔑 一句话总结 UserData 在每台节点首次开机时自动完成 4 件事：挂数据盘 → 装本地盘自愈服务 → 装 EFA 用户态 → 配 GPU 容器运行时。 执行约 3-5 分钟，全部日志在 /var/log/p5en-bootstrap.log。 完成后节点即具备：持久数据盘 /data、28TB 高速本地盘 /mnt/nvme（容器数据自动落这里）、16 张 EFA 网卡可用、docker/containerd 可直接跑 GPU 容器。

1. 四个阶段做了什么

[1/4] EBS 数据盘 → /data（持久存储）

做什么在 NVMe 总线上按型号 Elastic Block Store 找到那块空白的 2TB EBS 数据盘（跳过已有分区的根盘），格式化 xfs，挂载到 /data，按 UUID 写入 fstab
为什么模型分发中转、日志、JIT 缓存等需要重启不丢的数据放这里。stop/start 后自动重挂，数据保留
结果/data ~2TB（含 models/ logs/ cache/ 三个预建目录），持久

[2/4] 本地盘自愈服务（本模板的核心设计）

做什么①带重试安装 mdadm（网络未就绪时最多等 5 分钟）②安装 systemd 服务 setup-local-disks.service 并立即执行一次
服务干什么把 8 块 3.5TB 本地 NVMe（instance-store）组 RAID0 → 28TB ext4 → 挂 /mnt/nvme，然后把 /var/lib/docker 和 /var/lib/containerd bind 到 /mnt/nvme 上——容器镜像/层全部落高速本地盘，不占也不拖慢根盘
为什么是"服务"而不是一次性脚本本地盘物理绑定宿主机，stop/start 后是一组全新空盘（RAID 和文件系统全没了）。做成每次开机都跑的 systemd 服务（排序在 docker/containerd 之前），重启后自动重建 RAID + 重新 bind——自愈。盘上数据本身是临时的（丢失可接受），但磁盘布局永远自动恢复
两种重启的区别stop/start（换宿主机）：全新空盘 → 服务重建 RAID（设备名 md0），盘上数据丢失；reboot（软重启，不换宿主机）：盘和数据都还在，内核开机时会把阵列自动组装成 md127，服务探测并直接复用该阵列（不重建不格式化）——reboot 后 /mnt/nvme 数据保留。所以看到设备名是 md0 或 md127 都正常
安全设计fail-fast：mdadm 缺失、或 /mnt/nvme 挂的不是 RAID 设备时服务直接失败退出（状态可见可告警），绝不静默把容器数据写到根盘
结果/mnt/nvme ~28TB（临时，重启清空但自动重建）；docker/containerd 数据目录已 bind

[3/4] EFA 用户态（固定版本 1.48.0）

做什么下载并安装 aws-efa-installer 固定版本（--skip-kmod：内核模块用 DLAMI 自带的，只装用户态 libfabric）
为什么宿主机与推理容器内 EFA 版本必须对齐（镜像内同为 1.48.0）；固定版本保证节点可重复构建
结果/opt/amazon/efa，fi_info -p efa 可见 16 个设备

[4/4] GPU 容器运行时

做什么确认 nvidia-container-toolkit，把 nvidia 注册为 docker 和 containerd 两者的默认运行时并重启服务；nvidia-smi -pm 1 开持久模式
为什么无论后续用 docker 直跑还是 K8s(containerd)，GPU 容器都开箱即用
结果docker run --gpus all ... 直接可用

2. 完成后的节点磁盘/网络布局

┌─ p5en.48xlarge 开机完成后 ──────────────────────────────────────────────┐
│                                                                        │
│  根盘 1TB EBS ─── / (OS, 不放业务数据)                                  │
│  数据盘 2TB EBS ── /data          ◄─ 持久: 模型中转/日志/JIT缓存        │
│  8× 3.5TB NVMe ── RAID0 /mnt/nvme ◄─ 28TB 高速; stop/start 清空+自动    │
│                   (md0 或 md127)      重建, reboot 数据保留             │
│                      ├─ bind → /var/lib/docker      (容器镜像/层)      │
│                      ├─ bind → /var/lib/containerd  (K8s 场景同理)     │
│                      └─ models/ (模型权重, 重启后需重新分发)            │
│                                                                        │
│  主 ENI ×1 (私有 IP, 经 NAT 出网) + efa-only ENI ×16 (RDMA, 节点间通信) │
│  GPU: 8× H200 (nvidia runtime 已注册 docker+containerd)                │
└────────────────────────────────────────────────────────────────────────┘

3. 客户需要知道的三件事

⚠️ ① 网络前提（开机前确认） p5en 这类多网卡机型开机时拿不到公网 IP（EC2 限制：API 拒绝显式指定且忽略子网 auto-assign）。 UserData 要下载 mdadm/EFA installer，因此节点所在子网必须经 NAT Gateway 出网（推荐）， 否则 UserData 网络步骤会失败。部署前跑 preflight-cr.sh 会自动检查这一项。
⚠️ ② /mnt/nvme 是临时盘（stop/start 后数据清空；reboot 保留） stop/start（控制台 Stop 再 Start，换宿主机）：磁盘布局自动重建，但盘上的数据不会回来——模型权重需重新从 S3 分发，容器镜像需重新拉取/导入。reboot（sudo reboot 或控制台 Reboot，不换宿主机）： 数据保留，无需重新分发。需要持久的数据始终放 /data（EBS）。
✅ ③ 开机后 1 分钟验证清单（每节点） 
sudo tail -5 /var/log/p5en-bootstrap.log        # 末行 "p5en bootstrap complete"
systemctl is-active setup-local-disks.service    # active (failed = 自愈服务拦截了异常, 看 journalctl)
grep -c ^/dev/md /proc/mounts                    # 3 (/mnt/nvme + docker + containerd; 设备名 md0 或 md127 均正常)
df -h /data /mnt/nvme                            # ~2T / ~28T
/opt/amazon/efa/bin/fi_info -p efa -l | grep -c efa   # 16
nvidia-smi -L | wc -l                            # 8
docker run --rm --gpus all public.ecr.aws/docker/library/hello-world  # 容器运行时 OK

4. 排障速查

现象原因处理
bootstrap.log 出现 waiting for network to install mdadm 多次后 FATAL子网无出网（无 NAT）修路由（NAT）后 dnf install -y mdadm && systemctl restart setup-local-disks.service
setup-local-disks.service 状态 failedfail-fast 拦截（mdadm 缺失 / RAID 未成）——这是设计行为，防止数据静默写根盘journalctl -u setup-local-disks.service 看具体原因，修复后 restart 服务
grep -c ^/dev/md /proc/mounts 少于 3bind 未完成同上，restart 服务后 docker/containerd 也 restart
reboot 后服务 failed、日志见 Device or resource busy脚本版本过旧（2026-07-07 前）：reboot 后内核已把阵列组装成 md127，旧脚本硬编码 md0 重建失败更新 setup-local-disks.sh 到最新版（已支持探测复用 md127）后 restart 服务；数据仍在盘上不会丢
fi_info 无输出EFA 安装失败或 SG 缺自引用规则查 bootstrap.log [3/4] 段；确认 SG 自引用 ingress/egress
stop/start 后 /mnt/nvme 有了但模型没了预期行为（换宿主机=新空盘；注意 reboot 不属此情况）从 S3 重新 sync 权重；镜像重新导入

p5en.48xlarge 自建 K8s 推理系统部署文档

SGLang PD 分离（Prefill/Decode Disaggregation）· Mooncake KV Transfer · UCCL-EP MoE all-to-all · 16× EFA · Capacity Block 批量开机 · 2026-07

0. 概要说明（Executive Summary）

本文档面向在 AWS p5en.48xlarge（8× NVIDIA H200 141GB，16× EFA，共 3200 Gbps）上， 通过 Capacity Block（CB）一次开 8 台、使用自建 Kubernetes（非 EKS）构建 SGLang PD 分离推理系统的客户。

0.1 目标拓扑

角色GPU 数节点数并行策略DeepEP 模式
Prefill32 卡4× p5enpp4 / tp8 / ep32normal（大 batch 吞吐优先）
Decode64 卡8× p5endp64 / ep64low_latency（逐 token 延迟优先）
✅ 验证路径：先 2P2D 验证部署，再切换到 4P8D 目标部署方案 先用 4 台机器跑 2 Prefill 节点 + 2 Decode 节点（本文第二部分第 8 章给出完整启动命令）， 验证 EFA / Mooncake KV 传输 / UCCL-EP all-to-all 三条链路全部打通后，切换到目标 4P + 8D 方案。

注意：2P2D → 4P8D 不是简单扩容，而是部署方案的切换——两者并行策略完全不同 （Prefill pp2→pp4，Decode tp16/dp16→tp64/dp64），意味着权重切分方式、KV pool 布局、 JIT 编译缓存全部改变，需要修改启动参数并对全部 P/D 服务做停服重新部署（见 §9）。 可以复用的是基础设施层：启动模板 / 安全组 / placement group / 镜像 / AMI 从第一天起 按最终规模设计，两套方案通用，切换时只增加实例数量。

1前置准备

1.1 购买 Capacity Block

# 1. 搜索可用的 CB 时段（8 台 p5en，按需调 duration）
aws ec2 describe-capacity-block-offerings \
  --region us-east-2 \
  --instance-type p5en.48xlarge \
  --instance-count 8 \
  --capacity-duration-hours 168 \
  --output table

# 2. 购买（用上一步返回的 CapacityBlockOfferingId）
aws ec2 purchase-capacity-block \
  --region us-east-2 \
  --capacity-block-offering-id cbo-xxxxxxxxxxxx \
  --instance-platform Linux/UNIX

# 3. 查看预留 ID（cr-xxxx）与 AZ —— 后续开机脚本需要
aws ec2 describe-capacity-reservations \
  --region us-east-2 \
  --filters Name=instance-type,Values=p5en.48xlarge \
  --query 'CapacityReservations[].{id:CapacityReservationId,az:AvailabilityZone,state:State,start:StartDate,count:TotalInstanceCount}' \
  --output table
⚠️ CB 注意事项
  • CB 预留有固定起止时间，开始前无法开机；到期时实例被自动回收——因此本文用固定数量的 EC2 实例而非 ASG（ASG 会在回收后徒劳补机报错）。
  • 子网必须在 CB 预留的 AZ 内，跨 AZ 开机直接失败。
  • 2P2D 测试阶段可只开 4 台（Terraform 改 instance_count=4 / CFN 改 InstanceCount=4），CB 内未用容量保持可用，扩容时把参数改回 8 再 apply 即可。

1.2 查询 AL2023 Deep Learning AMI

基础镜像使用 AL2023 Base Deep Learning AMI（含 NVIDIA 驱动、EFA 内核模块、Docker、nvidia-container-toolkit）：

aws ssm get-parameter --region us-east-2 \
  --name /aws/service/deeplearning/ami/x86_64/base-oss-nvidia-driver-gpu-amazon-linux-2023/latest/ami-id \

1.3 网络前提

  • 一个 VPC + 一个与 CB 同 AZ 的私有子网（子网可用 IP ≥ 32：每台实例主网卡占 1 个 IP，efa-only 网卡不占 IP）。
  • 私有子网需具备出网能力（NAT Gateway 或 VPC Endpoints），用于拉取 EFA installer、容器镜像、K8s 软件源。
  • 准备一台堡垒机（或使用 SSM Session Manager——节点 IAM 角色已含 AmazonSSMManagedInstanceCore，可免 SSH 直连）。

2.3 启动模板关键片段（节选自 main.tf，完整见文件）

resource "aws_launch_template" "p5en" {
  image_id      = var.ami_id
  instance_type = "p5en.48xlarge"        # CB 要求 InstanceType 写入 LT

  instance_market_options { market_type = "capacity-block" }
  capacity_reservation_specification {
    capacity_reservation_target {
      capacity_reservation_id = var.capacity_reservation_id
    }
  }
  placement { group_name = aws_placement_group.cluster.name }

  # 根盘 1TB gp3
  block_device_mappings {
    device_name = "/dev/xvda"
    ebs { volume_size = 1024  volume_type = "gp3"  encrypted = true }
  }
  # 数据盘 2TB gp3 拉满 IOPS/吞吐
  block_device_mappings {
    device_name = "/dev/xvdb"
    ebs { volume_size = 2048  volume_type = "gp3"
          iops = 16000  throughput = 1500  encrypted = true }
  }

  # 主网卡: card 0 / device 0, 纯 ENA（官方要求主网卡不能是 efa-only）
  network_interfaces {
    network_card_index = 0
    device_index       = 0
    interface_type     = "interface"
    subnet_id          = var.subnet_id
    security_groups    = [aws_security_group.node.id]
  }
  # card 0 / device 1: EFA-only —— 容易漏掉的一个! card 0 同样承载 EFA
  network_interfaces {
    network_card_index = 0
    device_index       = 1
    interface_type     = "efa-only"
    subnet_id          = var.subnet_id
    security_groups    = [aws_security_group.node.id]
  }
  # card 1..15 / device 0: EFA-only x15, 合计 16 个 EFA 设备
  dynamic "network_interfaces" {
    for_each = range(1, 16)
    content {
      network_card_index = network_interfaces.value
      device_index       = 0
      interface_type     = "efa-only"
      subnet_id          = var.subnet_id
      security_groups    = [aws_security_group.node.id]
    }
  }
}

2.4 userdata 做的四件事（userdata.sh.tpl）

步骤动作结果
[1/4]按 NVMe 型号 Elastic Block Store 识别 2TB 数据盘（区别于本地盘的 Instance Storage 型号），mkfs.xfs 并写 fstab/data（含 /data/models /data/logs /data/cache）
[2/4]8× 3.5TB 本地 NVMe 组 mdadm RAID0（chunk 512K）+ ext4，fstab 加 nofail（实例存储重启即失效）/mnt/nvme（~28TB 高速盘）
[3/4]固定版本安装 EFA userspace（installer 1.48.0，与容器镜像内版本一致）/opt/amazon/efa，fi_info -p efa 可用
[4/4]确认 nvidia-container-toolkit + Docker；nvidia-smi -pm 1 开启 persistence mode容器运行时就绪
⚠️ RAID0 数据无冗余、重启丢失 /mnt/nvme 是实例存储（stop/start 后数据清空），只放可再生数据：模型权重（可从 S3 重拉）、JIT 编译缓存。 需要持久化的（日志、配置、编译产物备份）放 /data（EBS）。RAID0 拼装逻辑参考 KevinZhao/efa-validation scripts/setup-nvme.sh。

4.6 实例拓扑检查（跨节点通信性能预判）

EFA 跨节点延迟与实例在数据中心的物理拓扑相关。用 DescribeInstanceTopology API 查看各节点的网络层级（NetworkNodes 三层结构），尽量保证 Layer 1 & Layer 2 相同—— CB + cluster placement group 通常已保证邻近，此检查用于确认：

aws ec2 describe-instance-topology --region <R> \
  --filters Name=instance-type,Values=p5en.48xlarge \
  --query 'Instances[].{id:InstanceId,az:AvailabilityZone,nodes:NetworkNodes}' \
  --output json
# NetworkNodes[0](Layer1) 和 [1](Layer2) 相同的实例互访延迟最低。
# 若个别节点 Layer2 不同, 建议把它安排为 Prefill 节点(计算密集, 网络非瓶颈),
# Decode 节点间(MoE 小消息累积, 对延迟敏感)优先放拓扑最近的一组。
❌ 常见开机失败
  • InsufficientInstanceCapacity/InstanceLimitExceeded：CB 未到开始时间、或 cr-id/AZ/实例数与预留不符。
  • 网卡报错 InterfaceType not supported：子网 AZ 与 p5en 可用区不符，或网卡索引组合不合法（合法组合仅：card0/dev0=interface、card0/dev1=efa-only、card1-15/dev0=efa-only）。
  • EFA 设备只有 15 个而非 16：启动模板漏了 card 0 / device 1 的 efa-only 接口。
  • fi_info 无输出：安全组缺自引用 egress 规则，或 EFA userspace 安装失败（查 bootstrap 日志 [3/4] 段）。

4.5固化自定义 AMI：CB 换块后快速开机

🔑 为什么必须打 AMI Capacity Block 有固定到期时间，到期实例被自动回收，本地数据全部丢失。 下一个 CB 块开始时若从原始 DLAMI 重来，每台节点要重复：EFA 安装、K8s 组件安装、 11-12GB 推理镜像拉取导入、EFA device plugin 镜像导入……单节点耗时 30 分钟以上。 把配置完成的节点打成自定义 AMI，换 CB 块后用新 AMI 开机，节点分钟级就绪。

4.5.1 打镜像的最佳时机

在 §6 K8s 组件安装完成、§5 推理镜像已导入 containerd 之后、kubeadm init/join 之前（或从集群中挑一台已 join 的节点先 kubeadm reset 再打）。这样 AMI 里固化的内容：

4.5.2 打镜像操作

4.7磁盘方案设计边界与竞态知识（知识沉淀）

本方案让 docker/containerd 数据目录不落根盘的实现，是 setup-local-disks.service （systemd Before=docker/containerd）把空目录 bind 到本地 NVMe RAID0（/mnt/nvme）。 业界另有一条参考路线——eks-enterprise-quickstart 的 EBS 数据盘 + LVM + rsync 迁移方案， 其代码注释中记录过一次真实竞态事故。本节沉淀两条路线的边界结论，不涉及任何脚本改动。

4.7.1 为什么本方案没有 containerd 迁移竞态

eks-quickstart 记录的事故链：AMI 自带的 containerd 开机先在根盘启动 → 旧流程 stop 时未等 boltdb 落盘、就对正在变化的 content store 做 rsync（错误被 || true 吞掉） → metadata.db 引用的 blob 文件未真正落到 EBS → 少数节点 blob not found / ImagePullBackOff。该竞态需要三个条件同时成立，本方案一个都不满足：

竞态成立条件（eks-quickstart）本方案（v11）
containerd 已在根盘启动、boltdb 有未落盘数据重启路径：systemd Before= 排序保证 daemon 在 bind 完成前不可能启动；首启路径：systemctl stop（同步等待）后才 bind
对"活的" content store 做 rsync 迁移不迁移——bind 的是空目录，根盘旧数据只是被遮蔽（shadow），不存在 torn-copy
空 store 致命：EKS AL2023 AMI pin sandbox_image=localhost/kubernetes/pause，本地引用不可重拉（awslabs/amazon-eks-ami#2000 / #2122）DLAMI + kubeadm 自建，pause 来自 registry.k8s.io 可重拉；业务镜像本就走 ctr import 流程，空 store 是安全起点
🔑 结论 eks-quickstart 的 mask containerd + EXIT trap + rsync fail-fast 全套重手段， 是它的特殊前提（EKS AMI containerd 自启 + pause 不可重拉）逼出来的； 本方案靠"daemon 启动前完成 + 空目录 bind"从结构上绕开了竞态，不需要、也不要把那套复杂度引入。

4.7.2 适用边界：方案依赖 instance-store，本项目仅 p5en 故成立

机型本方案覆盖情况
p5en.48xlarge（本项目全部节点）✅ 覆盖：8x 3.5TB 本地盘组 RAID0；真机已验证（含 stop/start 自愈）
CPU d 机型（m6id/c6id 等，未来若加）✅ 天然覆盖，脚本零修改（单盘也可组 RAID0 + bind）
CPU EBS-only 机型（m6i/c6i 等，未来若加）❌ 不覆盖：脚本发现无本地盘直接 exit 0，docker/containerd 落根盘——镜像层/容器日志写满根盘 → kubelet DiskPressure 驱逐、节点 NotReady；若是专职 master 还与 etcd 抢根盘 IOPS。对策：优先换 d 机型；确需 EBS-only 机型时移植 eks-quickstart 的 EBS+LVM 方案（届时 mask/trap/rsync-fail-fast 缺一不可，否则即复现其事故）

4.7.3 静默退化路径：2026-07-05 真机踩中，已修复（fail-fast + NAT 出网 + mdadm 重试）

⚠️ 事故链（实测复盘） p5en 这类多网卡启动模板会忽略子网的 MapPublicIpOnLaunch → 公有子网里主 ENI 仍无公网 IP、开机初期完全无出网 → dnf install mdadm 拉 repo 超时 （AL2023 GPU DLAMI 不预装 mdadm）→ 旧脚本 || true 吞掉错误 → RAID 未建、bind 把根盘目录挂到 /var/lib/docker|containerd、 服务状态仍 active——数据静默落根盘。

已落地的三层修复（v12 当前版本）：

层修复位置
网络（07-06 修正）多网卡启动禁止 AssociatePublicIpAddress（EC2 API 400，真机 ROLLBACK 实锤）且忽略子网 auto-assign → p5en 开机不可能有公网 IP。标准方案 = 私有子网 + NAT Gateway（次选 = 开机后手工挂 EIP）；preflight-cr.sh 新增出网路由检查（默认路由 igw-* 直接 FAIL）模板注释固化 ×4 份 + preflight

脚本 fail-fastmdadm 缺失即 exit 1；bind 前校验 findmnt -no SOURCE /mnt/nvme == /dev/md0，不符即 exit 1——服务进 failed 状态，可见可告警，杜绝静默落根盘scripts/setup-local-disks.sh（base64 已同步重新内嵌）

4.7.4 reboot 路径：2026-07-07 真机踩中，已修复（探测复用已组装阵列）

⚠️ 事故链（实测复盘） reboot ≠ stop/start。软重启（OS 内 reboot 或控制台 Reboot）不换宿主机， 本地盘保留 RAID 超级块 → 内核在自愈服务运行前就把阵列自动组装成 /dev/md127 → 旧脚本硬编码 md0 且假设空盘，mdadm --create /dev/md0 报 Device or resource busy → fail-fast 校验正确拦截（挂载源非 md0，拒绝 bind）→ 服务 failed。保护生效（未静默落根盘），但本地盘整轮未被使用， docker/containerd 跑在根盘。

修复（v12 当前版本，真机验证 4/4）：脚本在组建前先探测这些 instance-store 盘是否已属于某个 md 设备（lsblk -rno NAME,TYPE 查 raid* 类型），已有则直接复用该设备名 （如 md127）挂载 + bind；仅全新空盘（stop/start 换宿主机）才走 create md0。 fail-fast 校验同步改为对比探测到的实际设备。两条路径行为：

场景宿主机本地盘数据脚本行为
stop/start更换全丢（新空盘）create md0 → mkfs → 挂载 → bind
reboot不变保留（同一组盘）复用内核已组装的 md127 → 直接挂载 → bind（不 mkfs，数据仍在）

6自建 Kubernetes 集群（kubeadm + EFA 适配）

流程来自本地已验证的 ec2-self-k8s-GPU-kubeflow.skill（自建 K8s + NVIDIA/EFA device plugin，含 12 条踩坑记录）。 该 skill 在 Ubuntu 24.04 上验证；本项目基础 AMI 是 AL2023 DLAMI，包管理相应从 apt 换成 dnf/rpm，其余逻辑一致。

🔑 K8s 网络设计决策
  • RDMA 流量与 CNI 无关：NCCL/Mooncake/UCCL 通过 aws-ofi-nccl / libfabric 直接走 EFA 设备，完全绕过 Pod CNI 网络。CNI 只承载控制面 TCP（rendezvous、K8s Service、router 分发）。
  • 本项目所有推理 Pod 用 hostNetwork: true：PD 分离要求 Pod 间以节点 IP 互访（dist-init-addr、bootstrap 端口），host 网络最直接且免除 CNI 性能层。
  • CNI 选择：跑 Flannel 即可满足 kube-system 组件通信；若要求 Pod 用 VPC 原生 IP 可切换 AWS VPC CNI，但必须设 MAX_ENI=1 + WARM_ENI_TARGET=0，否则 CNI 会向 EFA ENI 分配 secondary IP 导致跨节点 Pod 失联（skill 已踩坑 #9）；且移除 Flannel 后必须把 iptables FORWARD 策略改回 ACCEPT（踩坑 #10）。

net.bridge.bridge-nf-call-iptables  = 1

EOF

# 3. containerd: SystemdCgroup + NVIDIA runtime

sudo mkdir -p /etc/containerd
containerd config default | sudo tee /etc/containerd/config.toml > /dev/null
sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml

sudo systemctl restart containerd

9切换到目标部署方案：Prefill 32 卡（pp4/tp8/ep32）+ Decode 64 卡（dp64/ep64）

⚠️ 这是部署方案的切换，不是扩容 2P2D 与 4P8D 的并行策略完全不同（Prefill pp2→pp4、Decode tp16/dp16→tp64/dp64）， 切换意味着：
  • 权重切分方式改变：pp4 每个 stage 的层数与 pp2 不同，tp64/dp64 的 expert/attention 分布与 tp16/dp16 不同——服务启动时按新策略重新加载切分；
  • KV pool 布局改变：dp64 单 rank 容量与 dp16 完全不同（见 §9.4）；
  • JIT 编译缓存全部失效：新并行形状触发 DeepGEMM / Triton 重新编译（冷启 10-20 分钟，缓存目录可保留但新形状命中率为零）；
  • 全量停服重启：所有 P/D 服务按新参数重新启动、重新组集群（dist-init 组从 nnodes 2 变为 4 和 8），router 重新挂载。
可复用的部分：基础设施（LT/SG/PG）、镜像、AMI、K8s 集群本体不变—— 新节点开机（改 instance_count）→ kubeadm join → 镜像预热 → 权重分发，然后按 §9.2/9.3 的新参数全量重新拉起。

9.4 dp64 与请求长度的容量关系

DP attention 下单个请求只落在一个 DP rank（不跨 rank 切分），单请求最大输入长度受 单 rank KV pool 限制而非集群总量。eks-h200-deepseek-v4-pd.skill 在 dp16 实测中单 rank max_total_num_tokens≈44800 即为单请求上限。dp64 时单 rank pool 更小，长文场景需实测 max_total_num_tokens（启动日志打印）并据此设定业务侧输入上限，或降低 dp 换取更大单 rank pool。

9.5 更长期的扩展路径：机型升级与整组复制

💡 建议：超过 750B 的模型使用 p6-b300（B300）

p5en 单节点 8× H200 = 1.128TB HBM。750B 级 FP8 权重（如 GLM-5.1 754GB）加载后剩余 显存已非常紧张，长 context / 大 batch 下 KV cache 空间不足，只能靠收缩 context、压低并发来换稳定。 p6-b300.48xlarge（8× B300 288GB = 2.3TB/节点，约 2 倍于 p5en）同样权重加载后 KV cache 余量宽裕，长文本推理（120K+ 输入）场景收益显著。

换 B300 需注意的软件栈差异（来源 efa-validation BUILD_MATRIX B300 stack）：

  • SGLang 需 ≥ 0.5.12.post1（首个官方支持 B300 的版本），镜像用 Dockerfile.customer-b300 构建（非 h200 版）
  • attention backend 必须 --attention-backend flashinfer（不能用 fa3，无 sm_103 支持）
  • UCCL 需含 PR #950（修 B300 上 16 EFA 双 NIC/GPU 的选卡逻辑，否则 proxy init 直接 SIGABRT）
  • 启动模板网卡布局不同：p6-b300 为 card 0 纯 ENA（不承载 EFA）+ card 1-16 各一个 EFA； p5en 的 card 0 是 ENA(dev0)+EFA(dev1) 双接口——B300 的 card 0 上没有 dev1 EFA
  • Mooncake TE 版本不变（v0.3.11.post1），构建时加 CUDA_ARCHITECTURES="100;103"
💡 负载均衡 / 数据并行扩容：机器数量翻倍（8 → 16），其他都不变

单个服务组（4P + 8D）的吞吐达到上限后，横向扩容方式是整组复制：再开一组完全相同的 Prefill + Decode 集群，机器从 8 台翻倍到 16 台。所有配置零改动——启动模板、镜像、 SGLang 启动参数、并行策略（pp4/tp8/ep32 + dp64/ep64）全部原样复用，只需：

  • CB 购买量 / IaC 实例数翻倍（instance_count 8 → 16）
  • 第二组用独立的 dist-init-addr（两组互不感知，无跨组通信）
  • router 层做请求分发：sglang_router 支持多组 --prefill/--decode 地址， 或在两个 router 前挂一层 NLB/ALB 做负载均衡

整组复制优于继续加大单组并行规模（如 dp64→dp128）：不引入新的通信半径和调参风险， 故障域隔离（一组挂掉另一组照常服务），容量线性可预期。

10.2 性能预期：EFA/SRD vs 自建 RoCE（迁移基准参考）

若客户从自建 ConnectX/RoCE 机房迁移，可参考 AWS Summit 2026《750B MoE 分离推理：从 RoCE 到 EFA 的全栈验证》 （GLM-5.1 750B / 256 Expert / Top-8，SGLang，2P2D 4× p5en H200，120K 长文本输入）实测结论设定预期：

指标RoCE → EFA 变化解读
TTFT（首 token）+9%Prefill 计算密集，网络非瓶颈
TPOT（每 token）+31%Decode MoE 每步 ~150 次小消息累积，单次差 ~28μs
Max ITL（尾延迟）−73%（434→117ms，好 3.7×）SRD 多路径喷射负载均衡，尾延迟显著优于 RoCE ECMP

要点：平均 token 延迟略高属预期（SRD 与 RoCE 协议路径差异），但尾延迟大幅改善—— 线上 SLA 通常由 P99 决定，长文本高并发场景整体体验更稳。且 EFA 无需 PFC/ECN 专用交换机配置，运维复杂度远低于自建 RoCE。
